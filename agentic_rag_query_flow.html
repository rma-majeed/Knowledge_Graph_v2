1
User Consultant types a query
"How has our recommendation on EV battery suppliers evolved across Toyota engagements from 2019 to 2023?"
This is a multi-hop query — it spans multiple documents, multiple years, and asks for a trend, not a single fact. Standard RAG would likely fail this.
↓
2
Master Agent Query classification and planning
The master agent (Google ADK) reads the query and makes three decisions before any retrieval happens.
Query type → multi-hop (spans years + entities)
Strategy → needs graph traversal + temporal filtering
Agents needed → Retrieval Agent + Reasoning Agent
This planning step is what separates agentic RAG from a pipeline. A pipeline always runs the same steps. The agent decides the right steps for this specific query.
↓
3
Query Rewriter Agent Expand into sub-queries
The single query is decomposed into targeted sub-queries to maximise coverage across retrieval methods.
Sub-query 1: "Toyota EV battery supplier recommendation 2019"
Sub-query 2: "Toyota battery supply chain LG CATL Panasonic"
Sub-query 3: "EV battery sourcing strategy automotive consulting"
Each sub-query hits different parts of the retrieval pipeline — some catch keyword matches (BM25), others catch conceptual matches (vector).
↓
4
Retrieval Agent Parallel retrieval across all methods
All three sub-queries fire simultaneously across three retrieval channels.
BM25
Catches "Toyota", "LG Energy", exact year mentions
Vector
Catches semantic concepts — "battery sourcing", "cell supplier"
Graph
Traverses Toyota → Supplier edges in KuzuDB across all docs
→ RRF fusion merges all ranked lists → 30 candidate chunks
→ BGE cross-encoder reranks to top 20 by true relevance
↓
The agentic loop — this is what makes it different
5
Quality Evaluator Score the retrieved context
The evaluator scores the 20 chunks against the original query on three dimensions.
Coverage
Are all years present?
Relevance
Are chunks on-topic?
Completeness
Enough to synthesise?
Path A — score too low → retry
↩ Loop back to Query Rewriter Agent
The evaluator sends specific feedback:
"Found 2019 and 2023 chunks but missing 2020–2022. Reformulate to target mid-period engagements."
Query Rewriter generates new sub-queries:
"Toyota battery supplier review 2020 2021"
"EV supply chain gap analysis mid period"
Retrieval runs again. Max 2–3 retries before fallback.
Path B — score sufficient → proceed
Quality threshold met — 18 relevant chunks covering 2019–2023
Context assembled within token budget, enriched text applied where available, then passed to generation.
↓
6
Reasoning Agent Multi-document synthesis
The reasoning agent receives the assembled context and does what a plain pipeline cannot — it explicitly connects findings across documents before writing the answer.
Internal reasoning: "2019 chunks recommend CATL as primary. 2021 chunks introduce LG Energy as alternative. 2023 chunks show diversification recommendation away from single supplier. Pattern = supply chain de-risking trend."
↓
7
LLM Generation Answer with citations
The LLM (Gemini / cloud model) receives the reasoning agent's synthesised context and generates the final answer with inline citations — not just raw chunks.
Output: structured answer tracing the evolution from single-supplier to diversified sourcing, with [Source: Toyota_EV_Strategy_2019.pdf, p4] [Source: Battery_Supply_Review_2023.pptx, slide 12] citations.
↓
8
Session Memory Store for follow-up queries
ADK session management persists the full conversation — query, retrieved chunks, and answer — so follow-up queries can reference this context directly.
Next query: "Which of those suppliers did we recommend for Honda?" — the agent knows "those suppliers" refers to CATL and LG Energy from the previous turn, without re-retrieval.
